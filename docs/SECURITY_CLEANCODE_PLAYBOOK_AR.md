# دليل التشغيل الآمن والنظيف للمشروع

هذا الملف يوضح طريقة العمل الصحيحة في المشروع (Frontend + Backend) وما يجب فعله وما يجب تجنبه، مع قواعد الأمان والأداء.

## 1) صورة عامة عن بنية المشروع

- الواجهة (Factory - Next.js):
  - صفحات عامة/تخطيط: تعتمد Server Components افتراضيا.
  - صفحات تفاعلية (لوحة التحكم): تعتمد Client Components عندما يوجد state أو hooks أو تفاعل مباشر.
  - التحميل/الأخطاء: عبر loading.tsx و error.tsx.
- الباك (NestJS + PostgreSQL + Prisma):
  - جميع المسارات الحساسة خلف JWT + Permissions Guard.
  - التحقق من المدخلات عبر ValidationPipe + DTOs.
  - الاستيراد الكبير يدعم المسار المتزامن/غير المتزامن مع Queue.

## 2) SSR / CSR / Static: متى نستخدم كل نمط؟

- استخدم SSR/Server Components عندما:
  - تحتاج SEO أفضل.
  - تريد تحميل بيانات موثوقة على الخادم قبل العرض.
  - تريد تقليل منطق المصادقة في المتصفح.
- استخدم CSR/Client Components عندما:
  - الصفحة تفاعلية بالكامل (نماذج، Drag&Drop، حالات UI فورية).
  - تحتاج hooks للمتصفح مثل useEffect/useState.
- استخدم Static عندما:
  - المحتوى شبه ثابت ولا يحتاج بيانات لحظية.

قاعدة عملية:
- لا تضع منطق أمني حرج فقط في CSR.
- اجعل القرار الأمني الأساسي (السماح/المنع) على الخادم دائما.

## 3) Clean Code: ما يجب فعله

- افصل المسؤوليات:
  - UI فقط في المكونات.
  - استدعاءات API في hooks/lib.
  - منطق التحقق والتحويل في service layer.
- استخدم أسماء واضحة للدوال والأنواع.
- اجعل رسائل الأخطاء مفهومة وقابلة للتنفيذ.
- لا تكرر نفس المنطق في أكثر من ملف؛ أنشئ helpers مشتركة.
- كل تغيير في سلوك endpoint يجب أن يترافق مع اختبار.

## 4) Clean Code: ما لا يجب فعله

- لا تخلط منطق العرض مع منطق الوصول للبيانات داخل component ضخم.
- لا تعتمد على parsing ضمني غير واضح لأنواع الملفات.
- لا ترسل أخطاء خام للواجهة بدون صياغة واضحة.
- لا تستخدم fallbackات صامتة تخفي فشل الأمان أو التحقق.

## 5) أمان المشروع: ما يجب فعله

- التزم بـ JWT HttpOnly cookies وعدم تخزين التوكن في localStorage للإنتاج.
- فعّل Guards على كل endpoint حساس.
- فعّل ValidationPipe + DTO على جميع المسارات.
- طبّق throttle على نقاط الدخول الحساسة.
- فعّل helmet + CORS مضبوط حسب البيئة.
- افحص الملفات المرفوعة من خلال:
  - الامتداد
  - MIME type
  - حجم الملف
  - المحتوى القابل للمعالجة

## 6) أمان المشروع: ما لا يجب فعله

- لا تثق في التحقق من الواجهة وحده.
- لا تسمح برفع أي ملف بدون whitelist صريح.
- لا تقبل SQL/raw queries بدون حاجة قصوى ومراجعة أمنية.
- لا تطبع أسرار البيئة في السجلات.

## 7) سياسة رفع الملفات (Imports)

المدعوم للاستيراد الجدولي:
- .xlsx
- .xls
- .xlsm
- .xlsb
- .ods
- .csv
- .tsv
- .txt
- .json

غير مدعوم للاستيراد الجدولي المباشر:
- .pdf
- .doc
- .docx

مهم:
- ملفات Word/PDF لا تضمن بنية صف/عمود موثوقة، لذلك لا تستخدم لاستيراد بيانات الموظفين/المخزون.

## 8) الأداء والسرعة (Nest + Prisma + PostgreSQL)

- استخدم count/groupBy/aggregate بدل تحميل كل الصفوف للحسابات الإحصائية.
- أضف فهارس (indexes) على الحقول المستخدمة في where/orderBy المتكرر.
- استخدم pagination دائما في القوائم.
- للعمليات الضخمة:
  - تحقّق Dry-run أولا
  - ثم نفّذ asynchronously عبر Queue
- راقب زمن الاستجابة endpoints الحرجة في الاختبارات.

## 9) Checklist قبل أي Pull Request

- هل يوجد تحقق DTO كامل لكل مدخلات جديدة؟
- هل كل endpoint حساس محمي بـ JWT + Permissions؟
- هل أضفت/حدّثت اختبارات e2e أو unit للتغيير؟
- هل تأكدت أن السلوك الأمني على الخادم وليس الواجهة فقط؟
- هل يوجد تأثير أداء (استعلامات كثيرة، full scan، غياب index)؟
- هل رسائل الأخطاء مفهومة للمستخدم وفريق الدعم؟

## 10) أوامر مفيدة

- تشغيل الباك:
  - npm run start:dev
- بناء المشروع:
  - npm run build
- اختبارات الأمان الحالية:
  - npm run test:security
- اختبارات e2e كاملة:
  - npm run test:e2e
